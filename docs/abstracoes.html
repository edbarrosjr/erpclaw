Glue — Abstrações do catálogo

Três lentes além do treemap, aterradas no requires real e nas specs. Camadas = o que se apoia em quê (alicerce embaixo) · Value Streams = o que cruza domínios · Feature Model = o que pode ser combinado. 100% offline.
Pilha de dependência L0→L5: cada camada superior depende das de baixo; as setas de requires apontam só para baixo. O alicerce (L0/L1) sustenta tudo; verticais (L4) e sub-verticais (L5) ficam no topo. Largura/número = ações.
Os 4 processos end-to-end que atravessam módulos. Um único comando dispara efeitos cross-skill — ex.: submit-sales-invoice posta no GL, baixa estoque (SLE) e calcula imposto. Badges marcam os efeitos reais de cada estágio. Clique num estágio para ver as ações irmãs daquele passo (a funcionalidade completa do domínio). O P2P se bifurca em Material × Serviço (fatura avulsa, sem GRN/estoque).
Árvore de variabilidade (SPL): instalações válidas = Núcleo obrigatório + escolhas opcionais. Verticais são XOR na prática (um negócio escolhe 1). Clique num grupo para expandir. Abaixo, as restrições reais (requires/excludes).
mandatory sempre presentexor escolha exclusivaor qualquer combinação
O General Ledger não é uma camada nem um fluxo — é o ponto de convergência: todo domínio operacional posta no gl_entry, e sempre pelo mesmo portão validado (gl_posting / invariantes — ninguém escreve direto). Cada raio mostra os voucher_type que aquele domínio transforma em lançamento. Clique num domínio para ver seus documentos.
Não confunda o nome: gl_entry é a tabela (o livro); o raio ⚙ gl é o domínio que administra o razão (plano de contas, fechamento) e posta só ajustes (revalorização cambial, fechamento de exercício) — os outros 10 postam documentos de negócio.
